# Training Manual — Demo project

This manual is an **example**, shipped with the Claude Project Kit as a preview. It shows what a healthy `training-manual.md` looks like after a project has been running for a few weeks. Browse it to see the style; your own manual will evolve differently.

## How to read this manual

Entries fall into four shapes:

- **Lesson** — a principle learned the hard way (or near-miss prevented).
- **Insight** — a conceptual framing that unlocks a problem.
- **Technique** — a how-to or decision heuristic.
- **Pattern** — a reusable structural template.

Each entry is timeless: it stands alone without the logbook.

## Topic: Process

### Lesson: Codify as you build, not at the end

End-of-project documentation is always written too late, after the real-time context has faded. Instead, treat every working session as an opportunity to ask: "what, if anything, from today would help someone who joins this project in six months?" If the answer is something, write it down now.

This is the whole reason this manual exists. If you only remember one entry from this tour, remember this one.

### Insight: The bar is "would they thank me for this"

You will be tempted to add every observation to the manual. Resist. A weak entry does not just waste space — it degrades the signal of the strong entries around it.

Ask yourself: "would a new team member reading this in six months thank me for adding it?" If yes, add it. If you are unsure, the answer is no. Not every session produces a manual entry. Most produce none. That is correct.

### Technique: Capture failures on purpose

The most valuable entries in this manual — and in `experiments/` — are the ones that describe something that **did not work**. Negative examples are expensive to rediscover. Future-you will thank present-you for writing down "we tried X, it failed because Y, here is what we learned" even when the failure felt minor at the time.
